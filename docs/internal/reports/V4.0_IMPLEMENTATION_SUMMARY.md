## 2. 图/状态机执行引擎 (StateGraph + Checkpoint)

### 2.1 核心模块

| 文件 | 说明 |
|------|------|
| `execution/state_graph.py` | 轻量 StateGraph 执行引擎，参考 LangGraph Pregel 模型 |
| `execution/checkpoint.py` | Checkpoint 持久化抽象 + Memory/SQLite 实现 |

### 2.2 StateGraph 特性

- **节点**: async callable，接收 state 返回 partial_state_update
- **静态边**: `add_edge(source, target)`
- **条件边**: `add_conditional_edges(source, condition, {key: target})`
- **并行分支**: `add_parallel_edges(source, branches, merge_node)`
- **Reducer**: `set_reducer(key, reducer_func)` 支持列表追加等自定义合并
- **编译验证**: 检查节点存在性、孤立节点警告
- **执行循环**: 执行节点 → merge_state → checkpoint → determine_next
- **重放恢复**: `areplay(thread_id, from_node, checkpointer)` 从任意 checkpoint 继续
- **元数据**: `__thread_id__`, `__iteration__`, `__history__`, `__error__`, `__truncated__`
- **Mermaid 导出**: `to_mermaid()` 可视化流程图

### 2.3 Checkpoint 后端

| 后端 | 适用场景 |
|------|----------|
| `MemoryCheckpointer` | 开发/测试，进程内内存 |
| `SqliteCheckpointer` | 单实例生产级持久化 |
| `PostgresCheckpointer` | 多实例分布式 (接口预留) |

### 2.4 安全集成设计

- Checkpoint metadata 自动注入 `tenant_id`
- 支持从 pre-execute checkpoint 重放，确保恢复状态一致性

## 3. 多 Agent 原生编排 (Supervisor-Worker + MessageBus)

### 3.1 核心模块

| 文件 | 说明 |
|------|------|
| `agents/message_bus.py` | asyncio Queue-based pub/sub + 点对点直接发送 |
| `agents/worker.py` | Specialist Worker Agent，内部可持有 StateGraph 子图 |
| `agents/supervisor.py` | Supervisor Agent，任务分解 + Worker 匹配 + 结果聚合 |
| `agents/crew.py` | AgentCrew 编排器，封装 Supervisor + Workers |

### 3.2 架构设计

```
User Task
    |
    v
SupervisorAgent (decompose_task)
    |
    +---> WorkerAgent A (analyst)   ---+
    |                                  |
    +---> WorkerAgent B (researcher) ---+---> 聚合结果
    |                                  |
    +---> WorkerAgent C (writer)     ---+
```

- **Supervisor-Worker 模式**: 集中路由 + 子图自治
- **两种执行模式**: `parallel` (并行分派) / `sequential` (串行流水线)
- **Worker 匹配**: 基于角色评分 (role match + goal keyword)
- **消息总线**: 
  - `subscribe(topic, handler)` / `publish(msg)` 广播
  - `register_agent(agent_id)` / `send_direct(recipient, msg)` 点对点
- **容错**: 子任务失败可停止后续执行（串行模式）

### 3.3 关键修复

- `SubTask` dataclass 添加 `role` 字段（与 `decompose_task` 中使用的 `role=` 对齐）
- `asyncio.wait()` 改为 `asyncio.create_task()` + `asyncio.wait()` 避免 coroutine 直接传入
- Supervisor 注册自身队列以接收 Worker 直接回复
- `_dispatch_subtask` 从 Supervisor 队列直接读取结果，替代轮询缓冲区

## 4. 容器化与多租户

### 4.1 容器化交付物

| 文件 | 说明 |
|------|------|
| `Dockerfile` | Python 3.12 slim，支持 CLI/Web/Worker 三种模式 |
| `docker-compose.yml` | nexus-web + nexus-worker + redis + nginx |
| `deploy/nginx.conf` | 反向代理 + WebSocket 代理 + 静态文件服务 |
| `deploy/k8s/namespace.yaml` | nexus 命名空间 |
| `deploy/k8s/deployment.yaml` | 2 副本 Deployment + 健康探针 |
| `deploy/k8s/service.yaml` | ClusterIP Service |
| `deploy/k8s/hpa.yaml` | HPA: CPU 70% / Memory 80%，2-10 副本 |
| `deploy/k8s/pvc.yaml` | 10Gi 持久化卷 |
| `deploy/k8s/configmap.yaml` | 非敏感配置 |

### 4.2 多租户隔离

| 文件 | 说明 |
|------|------|
| `tenant/context.py` | `TenantContext` (frozen dataclass) + `TenantContextManager` (ContextVar) |
| `tenant/isolation.py` | `TenantRegistry` 内存配额管理 |

### 4.3 租户隔离策略

- **行级隔离**: 所有数据表添加 `tenant_id TEXT NOT NULL DEFAULT 'default'`
- **ContextVar 传递**: `TenantContextManager.set_current(ctx)` 在 async Task 中隐式传递
- **配额管理**: `max_agents`, `max_requests_per_min`, `max_concurrent_tasks`
- **MemoryStore 集成**:
  - `save(entry, tenant_id)` / `get_by_tenant(tenant_id)`
  - `save_checkpoint(session_id, state, tenant_id)` / `load_checkpoint(session_id, tenant_id)`
  - `search_fts(query, tenant_id)` / `search_vector(embedding, tenant_id)`
  - `tenant_metadata` 表存储租户配额和显示名称

### 4.4 数据库迁移

- `_ensure_column()` 自动为旧表添加 `tenant_id` 列
- `checkpoints` 表主键迁移: `session_id PRIMARY KEY` → `id AUTOINCREMENT` + `UNIQUE(session_id, tenant_id)`

## 6. 设计决策与权衡

### 6.1 为什么不直接集成 LangGraph？

- **自研理由**: 
  1. 需要原生集成安全审查节点（GuardrailsNode/TrustScoreNode）
  2. 避免对 LangGraph 的深度依赖（社区碎片化风险）
  3. 保持代码轻量可控（~500 行 vs LangGraph 数千行）

### 6.2 为什么选 Supervisor-Worker 而非 AutoGen GroupChat？

- **可控性优先**: 集中路由便于审计和干预
- **Worker 子图自治**: 每个 Worker 内部可运行独立的 StateGraph
- **消息总线解耦**: 未来可替换为 Redis/RabbitMQ 实现分布式

### 6.3 多租户为什么默认行级隔离而非 Docker 沙箱？

- **性能**: 行级隔离开销最小
- **OpenHands 原则**: "sandboxing should be opt-in"
- **扩展性**: 高风险任务可额外启用 Docker/Kata 沙箱

## 7. 已知限制与后续优化

| 限制 | 优化方向 |
|------|----------|
| StateGraph 单线程执行 | 未来支持真正的并行分支（asyncio.gather） |
| MessageBus 仅内存实现 | 接入 Redis Streams 实现跨进程消息 |
| Worker 匹配基于规则 | 接入 LLM 进行智能任务分解和 Worker 匹配 |
| TenantRegistry 仅内存 | 接入 PostgreSQL 实现持久化租户状态 |
| 容器化未包含 CI/CD | 添加 GitHub Actions 自动构建镜像 |
